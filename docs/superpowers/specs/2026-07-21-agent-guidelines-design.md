# Fast-Drone-250 Codex 任务准则设计

## 目标

在仓库根目录创建 `AGENTS.md`，用于约束后续 Codex 对 Fast-Drone-250 源码的分析、修改和验证。准则以提升 RK3588 机载部署后的飞行稳定性为首要目标，同时适配“WSL2 修改、Git 同步、机载端拉取并编译”的工作流。

## 文件定位

`AGENTS.md` 是仓库级长期协作规则，不重复大篇幅描述源码架构。以下三份文件作为背景索引：

- `1_workspace_structure.md`：工作区目录和模块定位。
- `2_topics_packages.md`：ROS 包、节点、话题和数据流。
- `3_planner_framework.md`：规划器二次开发方向、阶段和优先级。

三份文件仅供参考，可能存在错误、过时内容或尚未实现的设计。任何结论都必须用当前分支的源码、launch/YAML、`CMakeLists.txt`、`package.xml`、日志和测试结果复核。若发生冲突，以实际代码和可复现证据为准。

## 准则结构

`AGENTS.md` 将包含以下内容：

1. 项目目标与适用范围。
2. 参考文档及其证据优先级。
3. 修改前的上下文检查要求。
4. 面向飞行稳定性的设计原则：保守默认值、功能开关、失败回退、参数化、可观测性和避免过度安全承诺。
5. 规划、建图、轨迹执行和控制链路的重点检查项。
6. 编译、静态检查、仿真和回放验证的分级要求。
7. Git 工作流和交付报告格式。
8. 禁止覆盖用户已有修改、禁止无授权推送以及不得伪造验证结果等通用边界。

## 修改与验证工作流

后续任务应先确定涉及的数据流和包边界，再读取相关源码、头文件、launch/YAML、构建清单和调用方。修改应保持最小范围，并避免在一个补丁中混入无关重构。

验证按改动风险分级：优先执行目标包编译和相关测试；跨包接口、消息或公共头文件变化执行完整 Catkin 编译；涉及规划时序、地图、B 样条、轨迹发布或控制接口的变化，还应说明需要进行的仿真、rosbag 回放和机载端验证。无法执行的验证必须明确标注为未验证，不得以推测代替结果。

## Git 与部署边界

本地修改位于 `Fast-Drone-250-original`，用户后续通过 Git 将变更同步至机载电脑。Codex 可以检查状态和差异，但不应覆盖用户已有改动，也不得在未明确要求时执行 `git push`。每次交付应列出修改文件、原因、已运行的验证、未完成的验证、风险与回退方式。

顶层目录名称不作为运行时接口。准则中的命令应尽量从仓库根目录执行并使用相对路径，避免硬编码 `Fast-Drone-250` 或 `Fast-Drone-250-original`。

## 验收标准

- 文件名为仓库根目录下的 `AGENTS.md`。
- 内容简洁、可执行，并覆盖上述长期规则。
- 明确三份 Markdown 是参考而非事实来源。
- 不加入与 WSL2 修改环境无关的实机操作禁令。
- 不修改用户已有的构建产物或其他未跟踪文件。
- 完成后检查文件内容、`git diff` 和 `git status`，并报告目录改名的实际影响。
