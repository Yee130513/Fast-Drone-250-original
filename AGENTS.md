# Fast-Drone-250 源码优化任务准则

## 项目目标与工作方式

本仓库用于在 WSL2 中优化浙大 Fast-Drone-250 源码，最终通过 Git 将变更同步到 RK3588 机载电脑。所有修改以提高实机飞行的稳定性、可诊断性和可回退性为首要目标，同时保持 ROS1/Catkin 工程可编译、接口兼容和部署步骤可复现。

本文件适用于整个仓库。更深层目录若存在自己的 `AGENTS.md`，以距离目标文件最近的说明为补充；冲突时以更深层说明为准。
## 沟通与 Token 使用

- 默认采用简洁模式，只汇报结论、必要证据、修改文件和验证结果，不重复解释已经确认的背景。
- 优先进行定向搜索和局部读取，避免反复输出完整源码、完整日志、完整 Git 状态或无关第三方文件。
- 工具输出应主动限长；Git 推送、编译和测试默认不输出逐百分比进度或冗长成功日志，只保留摘要及错误上下文。
- 同一命令或检查不得无理由重复执行。命令失败时先判断失败原因，仅在重试能够产生新证据时重试。
- 普通问答、状态核对、简单配置和小型文档修改使用低开销分析；不为简单任务额外创建设计文档、实施计划或重复流程记录。
- 只有在用户明确要求详细分析/修改代码，或改动涉及飞行稳定性、控制、定位、规划、时序及故障回退等高风险链路时，才使用深入推理和更完整的验证。
- Token 节省不能替代安全验证。高风险修改仍须编译、测试或给出机载验证方案，但对用户只展示结果摘要和失败关键行。

## 参考资料与证据优先级

开始任务前，按需阅读以下背景资料：

- `1_workspace_structure.md`：工作区结构和主要源码位置。
- `2_topics_packages.md`：ROS 包、节点、话题及数据流。
- `3_planner_framework.md`：规划模块的优化设想、阶段划分和检查项。
- `readme.md`、`readme_en.md`：官方安装、调试和实验说明。

前三份文件是为了帮助 Codex 快速定位问题而补充的工程参考，内容可能不正确、过时，或描述尚未实现的方案，**仅供参考**，不得直接视为当前代码事实。

判断事实时使用以下优先级：

1. 当前分支的实际源码、头文件和消息定义。
2. 实际使用的 launch、YAML、`CMakeLists.txt`、`package.xml` 及脚本。
3. 可复现的编译结果、测试结果、rosbag、运行日志和话题观测。
4. 官方 README 和上游资料。
5. 上述三份补充说明及其他分析文档。

文档与代码冲突时，以代码和可复现证据为准，并在交付报告中指出冲突。不得为了符合参考文档而强行修改正确的现有实现。

## 修改前必须完成的检查

1. 先运行 `git status --short`，识别用户已有修改、删除项和未跟踪文件；不得覆盖、还原或顺带提交无关内容。
2. 明确问题所在的数据链路和 ROS 包边界，查清输入、输出、消息类型、坐标系、时间戳和调用关系。
3. 阅读目标 `.cpp`、`.h/.hpp` 及直接调用方；同时检查相关 launch、YAML、消息定义、`CMakeLists.txt` 和 `package.xml`。
4. 修改参数前确认参数的声明位置、默认值、ROS 参数名、单位、有效范围和所有使用点。
5. 修复 Bug 时先形成可验证的根因，区分定位、建图、规划、轨迹执行、控制和通信问题，不凭现象直接猜测。
6. 涉及参考资料中的 SCA-A*、动态安全阈值、ETM、非均匀 knot 或安全管理器时，先确认当前分支是否真实存在相应实现。

## 稳定性优先原则

- 保守默认：新算法或激进参数不得默认启用；首飞配置优先采用已验证的低速、低加速度和原有稳定路径。
- 可独立开关：实验性功能使用清晰的 ROS/YAML 参数独立控制，不把多个算法改动绑定到一个隐式条件。
- 可回退：搜索、优化、时间分配、轨迹复检或数据健康检查失败时，应保留明确的原始实现或保守结果作为回退路径。
- 失败闭合：不得发布未初始化、越界、包含 NaN/Inf、碰撞复检失败或动力学约束失败的轨迹与控制量。
- 连续性：重点检查位置、速度、加速度、yaw、时间参数和重规划衔接的连续性，避免指令跳变与控制流中断。
- 有界性：速度、加速度、jerk、knot 间隔、代价权重、阈值和迭代次数必须有合理边界；防止除零、数组越界、负时间和数值发散。
- 数据健康：深度图、VIO/里程计、地图和轨迹必须考虑时间戳、新鲜度、坐标系及异常输入，不能默认传感器数据始终可信。
- 可观测性：新增回退、超时、拒绝发布或降级分支时，提供节流日志或状态指标，并避免在高频循环中刷屏。
- 不夸大结论：除非有完整证明与覆盖性验证，不使用“绝对安全”“不会碰撞”“完全稳定”等保证性表述。

## 关键链路检查

### 感知、定位与建图

- 核对 RealSense 深度图、VINS/VIO 里程计的时间戳、坐标系、频率和超时行为。
- 修改 `grid_map`、raycasting、膨胀或 unknown-space 策略时，检查内存边界、地图分辨率、机体尺寸和传感器盲区。
- 不将 unknown space 静默当作 free；若设计确需如此，必须有显式参数、适用条件和验证记录。

### 搜索与轨迹优化

- 修改 A*/SCA-A* 步长、剪枝或稀疏扩展时，对候选边进行连续线段碰撞复检，并定义搜索失败回退。
- 修改 `base_point`、`direction`、clearance 或动态阈值时，检查关联连续性、梯度方向、上下界和无效 guide 的处理。
- 修改 B 样条控制点、时间分配或非均匀 knot 时，同时检查求值、导数、消息序列化、`traj_server` 解析和动力学约束。
- 新轨迹在发布前应按改动范围完成碰撞、速度、加速度、时间单调性和有限数值检查。

### FSM、轨迹执行与控制接口

- 修改重规划状态机时，逐项检查成功、失败、超时、连续失败和恢复路径，避免状态卡死或无界重试。
- `traj_server` 和 `/position_cmd` 相关改动应关注稳定发布频率、轨迹起止时间、末端行为和异常输入处理。
- 修改 `px4ctrl` 参数或控制接口时，保持消息语义、单位、坐标系和符号约定兼容；质量、悬停油门及增益不得使用未经确认的假设值。

## 实现要求

- 保持修改最小且聚焦，不混入无关重构、批量格式化或文件改名。
- 优先沿用现有 C++、ROS 和配置风格；公共接口变化必须同步检查全部生产者和消费者。
- 可调参数放入合适的 launch/YAML，并提供保守默认值；避免新增用户名、主目录或仓库名相关的绝对路径。
- 注释解释设计原因、单位、边界和回退条件，不重复代码表面行为，也不把未验证猜想写成事实。
- 不以删除检查、吞掉异常、固定返回成功或无限放宽阈值的方式让编译或测试“通过”。
- 不修改 `build/`、`devel/` 中的生成文件；源文件应在 `src/` 或相应配置目录中修改。

## 验证要求

根据风险选择最小但充分的验证，并记录真实结果：

1. 文档或脚本小改：运行格式/语法检查和相关只读检查。
2. 单个 ROS 包实现变化：优先编译目标包，例如 `catkin_make --pkg <package> -DCMAKE_BUILD_TYPE=Release`。
3. 消息、公共头文件、跨包接口或构建配置变化：从仓库根目录运行完整 `catkin_make -DCMAKE_BUILD_TYPE=Release`；也可使用仓库已有的 `sh shfiles/make.sh`。
4. 规划、建图、轨迹或控制逻辑变化：除编译外，优先增加单元/回归测试，并说明所需的仿真或 rosbag 回放场景、输入、指标和通过条件。
5. 无法在 WSL2 完成的机载依赖、性能或飞行验证，必须明确标为“未验证”，给出机载端命令、观察指标和停止/回退条件，不得推测为通过。

编译前注意 `build/` 和 `devel/` 可能保存绝对路径。仓库被移动或改名、工具链发生变化、缓存出现异常时，应建议用户在确认目标后进行干净重编译；未经允许不得删除现有目录。

## Git 与工作区规则

- 当前本地目录名可以是 `Fast-Drone-250-original`，命令和代码应从仓库根目录使用相对路径，不依赖顶层目录的固定名称。
- `origin` 用于用户仓库，`upstream` 用于浙大官方仓库。拉取或合并上游变更前先检查本地状态和冲突风险。
- 提交前检查 `git diff --check`、`git status --short` 和待提交差异，只暂存本任务文件。
- 未经用户明确要求，不执行 `git push`、改写历史、强制推送、删除分支或破坏性清理。
- 不提交 `build/`、`devel/`、ROS 日志、bag、临时文件、编辑器元数据、系统下载标记或大体积生成物，除非用户明确说明用途。
- 不擅自还原用户的已修改/已删除文件，也不使用 `git reset --hard`、`git clean` 或等价破坏性命令。

## 每次任务的完成报告

完成修改时必须报告：

- 修改了哪些文件及各自目的。
- 根因或设计依据，以及参考资料中哪些说法经过源码核验。
- 实际运行的编译、测试、静态检查或回放命令和结果。
- 未执行的验证、剩余风险和建议的机载端验证步骤。
- Git 提交状态；如已提交则给出提交哈希，如未提交则说明原因。
- 回退方法。优先说明如何回退本次提交或恢复相关参数，不覆盖其他用户改动。
